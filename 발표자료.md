## 1. 주제 배경

**왜 AI 목소리 탐지가 필요한가?**

- 2024년 보이스피싱 피해액 **8,545억 원** (경찰청)

  - Clova, ElevenLabs, DeeVid 등 고품질 TTS 서비스 무료 공개
  - 사람이 귀로 듣기엔 구분이 매우 어려운 수준
- 기존 보이스피싱 탐지: 텍스트(키워드) 기반 → TTS 목소리 자체를 탐지하는 기술 부재
- **→ 통화 중 실시간으로 AI 목소리를 자동 탐지하는 음성 레이어 필요**

## 3. 모델 선택 — RawNet2

**왜 RawNet2인가?**

- 기존 방식의 한계
  - MFCC, 스펙트로그램 등 특징값으로 변환 후 분류하는 방식
  - 변환 과정에서 TTS 특유의 고주파 아티팩트, 위상 패턴 등이 손실될 수 있음
  - 즉, 탐지에 핵심이 되는 정보가 전처리 단계에서 사라질 가능성

- RawNet2의 특징
  - **원시 파형(raw waveform)을 전처리 없이 직접 입력**
  - 어느 주파수 대역에 TTS 아티팩트가 있든 손실 없이 학습 가능
  - SincNet 필터뱅크(학습 가능한 대역통과 필터) → 잔차 블록(1D CNN) → GRU → FC 구조
  - 입력: `(batch, 1, 64600)` — 16kHz × 약 4초
  - 출력: `[진짜 logit, 가짜 logit]` → softmax → 확률

- 선택 근거
  - ASVspoof 2019 대회(AI 합성 음성 탐지 전문 대회)에서 검증된 모델
  - 우리가 하려는 태스크(TTS vs 실제 음성 구분)와 목적 완전 일치
  - 사전학습 가중치(ASVspoof 2019) 공개 → 처음부터 학습 불필요, 한국어 데이터 파인튜닝만으로 높은 성능 달성 가능

## 4. 학습 데이터 구성

### 데이터 분리 전략 — Speaker-based Split

- 단순 랜덤 분리의 문제점
  - 같은 화자의 음성이 train/dev 양쪽에 들어가면 모델이 화자 목소리 자체를 외워서 EER이 비정상적으로 낮게 측정됨 (데이터 리키지)
- 해결: holdout 화자 2명(`nkyunglee`, `njooahn`)을 dev 전용으로 완전 격리
  ```
  전체 샘플
  ├── holdout 화자 → dev 고정 (464개)
  └── 나머지 → 80/20 랜덤 분리
      ├── 80% → train
      └── 20% + holdout → dev
  ```

### Fake (가짜) — 총 11,000개+

| 데이터 | 수량 | 설명 |
|--------|------|------|
| AI Hub 음성합성 데이터 | ~10,000개+ | AI Hub "감성 및 발화 스타일별 음성합성 데이터" — 다양한 화자/감성/발화 스타일 |
| Qwen3-TTS | 240개 | 6개 환경 카테고리 × 40개 (train용) |
| speaker_TTS | 25개 | 특정 화자 TTS (별도 수집) |
| DeeVid | 일부 | 영상 기반 딥보이스 TTS |
| clova_tts | 1248개 | 딥보이스 TTS |
| clova_tts_mp3sim | 300개 | AI Hub 음성합성 파일을 MP3 인코딩→디코딩한 압축 시뮬레이션 버전 |

- Qwen3-TTS 6개 카테고리: clean / phone / compress / reverb / emotion / phone_compress
  - 실제 통화 환경에서 발생할 수 있는 다양한 음향 조건을 사전 반영

### 데이터 추가 배경 — 단계별 개선

**초기 문제 — EER 0% 과적합**

처음 학습을 시작했을 때 dev EER이 단 몇 epoch 만에 **0%에 가깝게 수렴**했습니다.

- EER 0%는 겉으로는 완벽해 보이지만, 실제로는 **과적합(overfitting)** 신호
- 원인 분석: 데이터셋이 너무 단순하고 소량 → 모델 입장에서 문제가 너무 쉬움
  - 한 종류의 TTS(AI Hub 음성합성)만 학습 → 해당 TTS의 아티팩트 패턴만 외움
  - 학습/검증 세트에 같은 화자 음성이 섞여 있어 화자 목소리 자체를 암기 (데이터 리키지)
- **해결 방향**: 모델이 다양한 상황에서 일반화할 수 있도록 TTS 종류와 진짜 음성을 **점진적으로 추가**
결론: 같은 화자가 train/dev 양쪽에 포함되어 있었고, 데이터가 너무 단순하여 EER 0에 빠르게 수렴
처음부터 모든 데이터를 넣은 것이 아니라, **탐지 실패 케이스를 발견할 때마다 해당 데이터를 추가하는 반복적 개선** 방식으로 데이터셋을 구성했습니다.

| 단계 | 추가 데이터 | 추가한 이유 |
|------|------------|------------|
| 1차 | AI Hub 음성합성 + 일반 한국어 음성 | 가장 구하기 쉬운 공개 합성음/실제음으로 기초 학습 |
| 2차 | Qwen3-TTS (6개 환경 카테고리) | 1차 학습 후 Qwen3 TTS가 진짜로 판별됨 → 다른 TTS 엔진도 학습 필요 |
| 3차 | speaker_TTS, DeeVid | 스피커 재녹음 TTS·영상 기반 TTS도 탐지 못함 → 추가 |
| 4차 | real_calls, news | 진짜 음성 다양성 부족, 실제 전화 통화 및 뉴스 추가로 진짜/가짜 경계 강화 |
| 5차 | clova_tts_mp3sim | DeeVid MP3 파일 탐지율 낮음 → MP3 압축 환경을 학습시키기 위해 추가 |

**1차 학습 탐지 결과 (데이터 고도화 이전)**

| 테스트 대상 | 결과 | 후속 조치 |
|------------|------|----------|
| AI Hub 음성합성 (Clova 계열) | ✅ 가짜 탐지 | — |
| Qwen3-TTS | ❌ 진짜로 판별 | Qwen3-TTS 6개 카테고리 추가 |
| 스피커 재녹음 TTS | ❌ 진짜로 판별 | speaker_TTS + reverb augmentation 추가 |
| DeeVid MP3 | ❌ 진짜로 판별 | clova_tts_mp3sim 300개 추가 |

→ **특정 TTS만 학습하면 다른 TTS는 탐지 못하는 문제**를 반복 실험을 통해 확인하고 해결했습니다.

## 5. Augmentation 전략

**학습 중 실전 환경을 데이터로 시뮬레이션**

| 기법 | 확률 | 목적 |
|------|------|------|
| Speed perturbation ±5% | 20% | 재생 속도를 변형한 TTS 대응 |
| Volume 조절 (0.7~1.3배) | 80% | 마이크 감도/거리 차이 대응 |
| Phone channel (16kHz→8kHz→16kHz) | Fake 75% / Real 40% | 전화 코덱 압축 시뮬레이션 |
| Background noise (SNR 18~30dB) | Fake 35% / Real 25% | 실전 주변 잡음 환경 |
| Reverb | 20% | 스피커로 출력 후 재녹음 시뮬레이션 |

- **Phone channel 설명**
  - 전화 통화는 16kHz → 8kHz로 압축 전송 후 다시 16kHz 복원
  - 이 과정에서 고주파 성분이 잘려나가고 TTS 아티팩트도 변형됨
  - fake에 75%를 적용한 이유: 보이스피싱 TTS가 대부분 전화로 전달되기 때문
  - 100%로 설정하지 않은 이유: 코덱 없는 VoIP/직접 재생 환경도 학습해야 하기 때문

- **핵심 원칙**: train에만 augmentation 적용, dev는 항상 클린 로드
  - dev에 augmentation을 적용하면 매 epoch마다 입력이 달라져 EER 수치를 신뢰할 수 없음

## 6. 성능 지표 — EER

**EER(Equal Error Rate)이란?**

- FAR(False Acceptance Rate): 가짜를 진짜로 통과시키는 비율
- FRR(False Rejection Rate): 진짜를 가짜로 차단하는 비율
- EER: FAR = FRR이 되는 지점의 비율 → 낮을수록 좋음
- EER 1%라면 → 가짜를 1%만 놓치고, 진짜를 1%만 잘못 차단하는 균형점

**학습 결과**

- 총 20 epoch 학습 (CosineAnnealingLR 스케줄러로 학습률 점진적 감소)
- 매 epoch 후 dev EER 측정, EER이 개선될 때만 `korean_model.pth` 저장 (best checkpoint 방식)

| Epoch | Loss | Accuracy | dev EER | 비고 |
|-------|------|----------|---------|------|
| 1 | 0.4297 | 86.76% | 5.86% | 학습 시작 |
| 2 | 0.1848 | 92.88% | 4.10% | |
| 3 | 0.1457 | 94.40% | 2.67% | |
| 4 | 0.1195 | 95.44% | 2.19% | |
| 5 | 0.1030 | 96.31% | 1.81% | |
| … | … | … | … | (6~19 epoch 지속 개선) |
| **최종 (best)** | | | **1.03%** | **20 epoch 중 최적 checkpoint** |

- 최종 EER **1.03%** → ASVspoof 2019 상위권 시스템 수준
- Accuracy **96%+** 달성
- train/dev 샘플 수: train 17,231 / dev 4,769 (holdout 462개 + 랜덤 4,307개)

## 7. 온디바이스 배포

**서버 없이 Flutter APK 안에서 직접 실행**

- 배포 방식을 온디바이스로 선택한 이유
  - 서버 왕복 없이 실시간 탐지 가능 (지연 최소화)
  - 개인 음성 데이터가 외부 서버로 전송되지 않음 (프라이버시)

- 변환 과정
  ```
  korean_model.pth (PyTorch)
      ↓ torch.onnx.export (opset 11)
  rawnet2.onnx
      ↓ onnx2tf 변환
  korean_model.tflite ← Flutter 앱에서 사용
  ```

- 변환 시 주의사항
  - PyTorch는 channels-first: `(batch, channel, time)` = `(1, 1, 64600)`
  - TFLite는 channels-last로 자동 변환: `(1, 64600, 1)`
  - 추론 시 입력 shape을 `(1, 64600, 1)`로 맞춰야 함

- 추론 방식 (슬라이딩 윈도우)
  - 모델 입력이 4초 고정이므로, 긴 통화는 4초씩 잘라서 반복 분석
  - 0.5초 간격으로 윈도우 이동, 무음 구간은 건너뜀 (RMS 에너지 기반 VAD)
